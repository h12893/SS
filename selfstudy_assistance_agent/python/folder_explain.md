各フォルダの概要

----------------------------------------------------------------------------------------------------

# .venv
pythonの仮想環境フォルダ。

# \_\_pycache\_\_
Pythonプロジェクトの実行中に自動生成される、Pythonのバイトコードキャッシュを格納する特別なフォルダ。

## 高速化 
Pythonが.pyファイルをインポートする際に、ソースコードをバイトコード（Python仮想マシンで実行可能な形式）にコンパイルし、.pycファイルとして保存。  
これにより、次回のインポート時に再コンパイルをスキップし、実行速度を向上させる。

## 自動生成
Python 3.2以降で導入された機能で、モジュールを初めてインポートすると自動的に作成。  
開発中や実行中にのみ存在し、ソースコードの変更があれば自動更新。

## 中身
主に.pycファイル（コンパイルされたバイトコード）が含まれる。  
ファイル名は通常、モジュール名.cpython-バージョン.pycの形式（例: connect_DB.cpython-313.pyc）。ここでは、Python 3.13でコンパイルされたconnect_DB.pyのバイトコードが保存されている。  
バイナリ形式のため人間が直接読むことはできないが、Pythonインタプリタが使用。

## バージョン管理からの除外 
.gitignoreファイルに__pycache__/を追加して、Gitなどのバージョン管理システムから除外し、リポジトリが不要なキャッシュファイルで汚染されるのを防ぐ。

## 削除
安全に削除可能（rm -rf __pycache__/ や手動削除）。次回の実行時に自動再生成される。

## 注意点
開発環境では残しておくのが一般的だが、本番環境や配布時には除去（例: python -m py_compile で明示的にコンパイル後、ソースを除去）。  
Pythonのバージョンが変わると古いキャッシュは無効化され、再生成される。

# config
処理実行時のパラメータなどを管理するconfigファイルの格納フォルダ。

# data
使用するデータの一時保存などの格納フォルダ。

# notebook
ipynbファイルの格納フォルダ。
ipynbファイルは、EDAや試行錯誤段階の処理を結果を確認しながら進める場合に使用。
処理内容が確定した本番や、繰り返し実行する内容などは.py形式で作成する。

# output
処理による出力結果の格納フォルダ。
出力内容に応じて、内部でさらにフォルダ訳をする（例：グラフ等の画像はimgなど）

# source
.pyファイルの格納フォルダ。
本番処理のファイルだけでなく、よく使う関数をまとめた.pyファイルも内部のmylibフォルダに格納。

## \_\_init\_\_.pyに関して
パッケージ（モジュールの集合体）を定義するファイル。  

### sourceフォルダの__init__.py
#### 役割
sourceフォルダをPythonパッケージとして認識させる。  
これにより、sourceをモジュールとしてインポート可能（例: import source や from source import something）。  
パッケージの初期化コードを含めることができる（例: 変数の定義、関数の実行、依存関係の設定）。  
空のファイルでもパッケージとして機能するが、初期化が必要な場合はここにコードを記述。  
Python 3.3以降では空の__init__.pyがなくてもパッケージとして扱われる場合があるが、明示的に置くのがベストプラクティス（互換性と可読性のため）。  

#### 中身
通常、空か、初期化コードのみ。  
sourceフォルダの場合、シンプルなパッケージ定義として機能。  

#### 管理方法
バージョン管理に含める（.gitignoreで除外しない）。  
変更時はパッケージの動作に影響するので、テストを推奨。不要なコードは避け、必要最小限に。  

### mylibフォルダの__init__.py
#### 役割
mylibフォルダをsourceパッケージ内のサブパッケージとして定義。  
これにより、mylib内のモジュール（例: connect_DB.py, make_output.py）をインポート可能（例: from source.mylib import connect_DB）。  
サブパッケージの初期化コードを含める（例: 共通のインポートや設定）。  
空でも機能するが、複数のモジュールをまとめて扱う場合に便利。  
パッケージ階層を明確にし、モジュールの整理を助ける。  

#### 中身
空のファイル。  
mylib内のモジュールを直接インポートするためのマーカーとして機能。

#### 管理方法
sourceフォルダと同様、バージョン管理に含める。  
サブパッケージの構造変更時は、インポートパスを確認。  
不要になったら削除可能だが、パッケージ構造を維持するために残すのが一般的。  

### 全体の管理Tips
#### 配置
各パッケージ/サブパッケージフォルダのルートに置く。  

#### ベストプラクティス
空の__init__.pyを置いてパッケージを明示的に定義。  
初期化コードが必要な場合のみ追加（例: __all__リストで公開モジュールを指定）。  

#### 注意
__pycache__フォルダと違って、バージョン管理から除外しない。  
Pythonの実行時に自動生成されるキャッシュとは異なり、__init__.pyは手動で管理。

## requirements.txt
環境にインストール中のライブラリとバージョンの一覧ファイル。
