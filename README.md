# 全体構成

## プロジェクトフォルダ

各フォルダの構成は以下の通り。必要に応じてそれぞれのフォルダ内部でさらにフォルダを分ける。

### 各フォルダ構成

- /docs：各種ドキュメントを格納したフォルダ。ドキュメントは特定の目的が無い限り.md形式で記述。

- /raw_data：プロジェクトに利用した元データを格納したフォルダ。コミット対象外。

- /adf：ADFのARMテンプレートと個別JSONファイルを格納したフォルダ。ARMテンプレートはコミット対象外。

- /sql：プロジェクトで利用したSQLクエリを格納したフォルダ。各ファイルは.sql形式で記述。

- /py：プロジェクトで利用したpythonファイルを格納したフォルダ。各ファイルは.py形式や.ipynb形式で記述。

- contents.md：プロジェクトフォルダ内の各フォルダ、ファイルの内容を記述したファイル。

**********

上記構成のプロジェクトフォルダとは別に.gitignoreファイルを作成し、コミット対象としないファイルやフォルダを指定。

# 運用ルール

## 更新作業

GitHub Flowに従って進める。

1. リモートリポジトリのmasterブランチをローカルリポジトリにpull
2. masterブランチから作業ブランチを作成
3. 作業後にローカルリポジトリに内容をコミット
4. 作業ブランチと同名のブランチをリモートリポジトリにプッシュ
5. リモートリポジトリでプルリクエスト送信
6. 内容を確認して問題なければmasterブランチにマージ
7. masterブランチをデプロイ（必要に応じて）

## タグ付与

タグは以下の**タグ付与条件**のいずれかに該当した場合に付与する。

### タグ付与条件

- 新規のプロジェクトを始めた場合

- 既存プロジェクトが完了した場合

- 新規のデータをデータベースに追加した場合

- アウトプットを伴う処理の追加、修正が発生した場合

タグは以下のコマンド、形式で作成する。

`git tag -a タグ名 -m "メッセージ"`

or 

`git tag -a タグ名*（エディタでメッセージ入力）`

- タグ名：yyyymmdd_概要（概要は〇〇プロジェクト開始、〇〇データ追加、〇〇処理修正など）

- メッセージ：注釈の概要の説明（1~2行程度）
