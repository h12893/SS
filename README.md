# 全体構成

## プロジェクトフォルダ

各フォルダの構成は以下の通り。必要に応じてそれぞれのフォルダ内部でさらにフォルダを分ける。

### 各フォルダ構成

- /ADF_info：ADFのARMテンプレートと個別JSONファイルを格納したフォルダ。ARMテンプレートと個別JSON内の/linkedServiceはコミット対象外。
- /docs：各種ドキュメントを格納したフォルダ。ドキュメントは特定の目的が無い限り.md形式で記述。
- /PowerBI：PowerBIによる可視化関連のファイルを格納したフォルダ。
- /raw_data：プロジェクトに利用した元データを格納したフォルダ。コミット対象外。
- contents.md：プロジェクトフォルダ内の各フォルダ、ファイルの内容を記述したファイル。

上記構成のプロジェクトフォルダとは別に.gitignoreファイルを作成し、コミット対象としないファイルやフォルダを指定。

### ADF_infoフォルダの更新に関して

ARMテンプレートの更新有無の判断は、**変更内容がADF全体の構造に影響するか否か**を基に行う（構造に影響する場合のみARMテンプレートを更新）。

**変更内容が構造に影響しない軽微な場合**は、個別JSONの更新、差分管理のみで十分。

ARMテンプレートの更新有無にかかわらず、個別JSONは全ての作業内容に対して更新を行う。

#### ARMテンプレート更新の例

- 新規パイプラインの追加
- 新規データセットの追加
- 新規リンクサービスの追加
- 新規トリガーの追加
- Integration Runtimeの追加、設定変更
- 既存パイプラインの大規模改修（処理フローが変わる）
- 既存データセットの構造変更（スキーマ変更など）
- リンクサービスの接続先変更（Key Vault の参照先変更など）
- トリガーのスケジュール変更（運用に影響する）

#### 個別JSON更新のみの例

- パイプライン内のログメッセージの修正
- コメントの追加、修正
- アクティビティ名の変更（処理内容に影響しない）
- UI上の並び順の変更
- 軽微なパラメータ名の変更（処理フローに影響しない）
- デバッグ実行のための一時的な設定変更

# 運用ルール

## リポジトリ更新作業

GitHub Flowに従って進める。

1. リモートリポジトリのmasterブランチをローカルリポジトリにpull
2. masterブランチから作業ブランチを作成
3. 作業後にローカルリポジトリに内容をコミット
4. 作業ブランチと同名のブランチをリモートリポジトリにプッシュ
5. リモートリポジトリでプルリクエスト送信
6. 内容を確認して問題なければmasterブランチにマージ
7. ローカル、リモート共に作業ブランチを削除
8. masterブランチをデプロイ（必要に応じて）

## タグ付与

タグは以下の**タグ付与条件**のいずれかに該当した場合に付与する。

### タグ付与条件

- 新規のプロジェクトを始めた場合
- 既存プロジェクトが完了した場合
- 新規のデータをデータベースに追加した場合
- アウトプットを伴う処理の追加、修正が発生した場合

タグは以下のコマンド、形式で作成する。

`git tag -a タグ名 -m "メッセージ"`

or 

`git tag -a タグ名*（エディタでメッセージ入力）`

- タグ名：yyyymmdd_概要（概要は〇〇プロジェクト開始、〇〇データ追加、〇〇処理修正など）
- メッセージ：注釈の概要の説明（1~2行程度）
